# Agent-DEMO 前端設計分析與優化建議

## 一、現況總覽

### 技術棧
- **框架**：Streamlit（單頁、側欄導航、chat 為主）
- **主題**：`.streamlit/config.toml` 深色 + 青綠主色、Plus Jakarta Sans / JetBrains Mono
- **自訂樣式**：`streamlit_app.py` 內 `_inject_custom_css()` 注入 CSS

### 頁面結構
| 區塊 | 內容 |
|------|------|
| 主標 | Agent-DEMO + 副標「RAG · 圖表 · 知識庫問答 · 多輪對話」 |
| 側欄 | 畫面切換（對話 / Eval 運行記錄 / Eval 批次結果）、設定、對話列表與操作 |
| 主內容（對話） | 歷史訊息、上傳/灌入文件展開器、chat_input |
| 主內容（Eval） | 篩選 + 列表/指標 + 展開詳情 |

---

## 二、現有優點

1. **主題一致**：深色 + 單一主色，專業感足夠。
2. **字型有辨識度**：非 Inter/Roboto，Plus Jakarta Sans 易讀。
3. **聊天區有層級**：訊息卡片圓角、使用者/助理視覺區隔、展開器統一風格。
4. **側欄分區清楚**：設定 / 對話 用 subheader + divider。
5. **主內容寬度受限**：max-width 900px 利於閱讀。

---

## 三、可優化面向與具體建議

### 1. 資訊架構與導航

| 問題 | 建議 |
|------|------|
| 「畫面」用 radio 三選一，切到 Eval 後主內容只剩標題+副標再+該頁，標題區在 Eval 頁略顯多餘 | **可選**：Eval 頁不顯示副標，或改為「檢視執行記錄」等情境化小標。 |
| 側欄「設定」裡 index/model 以 caption 呈現，字多時不易掃描 | 考慮用 `st.code(index_name, language=None)` 或縮短為「Index: xxx · dim=768」單行。 |
| 對話列表只有「未命名對話」時辨識度低 | 可依第一則使用者問題自動取前 20 字當對話標題（需在 append message 時更新 title）。 |

### 2. 視覺層級與留白

| 問題 | 建議 |
|------|------|
| 對話頁標題+副標+錯誤區+歷史+上傳區+輸入框 全在同一流，空狀態時較平 | 空對話時可加一句引導文案（例如：「在下方輸入問題，或先上傳文件灌入知識庫」）並稍微加大留白。 |
| Eval 運行記錄 / 批次結果 的「篩選」與「列表」之間關係可更明顯 | 篩選區用 `st.container()` 或輕量邊框包成一小塊，或加「篩選」小標。 |
| 展開器標題過長時（例如 Eval 那行）會折行或擠壓 | 列表項可改為：左側 icon（✓/✗）+ 主要資訊，次要資訊用 caption 或第二行。 |

### 3. 色彩與對比

| 問題 | 建議 |
|------|------|
| 僅有深色主題，部分使用者偏好淺色 | 在 config.toml 補 `[theme.light]`，讓使用者用 Streamlit 右上角切換；或維持單一深色並在說明文件註明。 |
| 成功/警告/錯誤仍為預設紅黃綠，可能與主色不夠協調 | 在 config.toml 設定 `redColor` / `orangeColor` / `greenColor`（或對應 background），與主色調統一。 |

### 4. 互動與回饋

| 問題 | 建議 |
|------|------|
| 灌入文件成功只靠 `st.success`，易被忽略 | 成功後可短暫顯示「已灌入 N 個 chunks，可直接在下方問答」並可選 2 秒後自動收合。 |
| 沒有「正在連線 / 初始化」的明確狀態 | 已有 `st.error` 初始化失敗；可選在 try 前加 `st.spinner("正在連線…")`，成功後不顯示。 |
| Eval 批次結果 選不同 run 時整頁重算，無載入提示 | 讀取 jsonl/metrics 前加 `with st.spinner("載入 Run…")`。 |

### 5. 響應式與可及性

| 問題 | 建議 |
|------|------|
| 主內容 900px 在超大螢幕會留很多空白 | 可接受；若希望更彈性可改為 `max-width: min(900px, 95vw)`。 |
| 側欄在窄螢幕會佔滿寬度，對話區被壓縮 | Streamlit 預設行為；可於說明中建議桌面使用。 |
| 對比度未明確檢查 | 主文 #F1F5F9 於 #0F172A 上已達 WCAG AA；caption #94A3B8 可再確認對比比。 |

### 6. 程式與維護面

| 問題 | 建議 |
|------|------|
| 自訂 CSS 全在字串裡，較長後難維護 | 可抽成 `assets/custom.css` 或專用 `.streamlit/custom.css`（若未來 Streamlit 支援），或至少按區塊註解分節。 |
| `data-testid` 為 Streamlit 內部用，未來可能變動 | 已知風險；若變動只需改 CSS selector，影響範圍小。 |

## 五、小結

目前前端已有清晰的主題、字型與聊天區樣式，主要可加強的是：**資訊層級**（Eval 篩選、空狀態）、**回饋**（載入中、灌入成功說明）、**主題延伸**（淺色、語意色）。以上建議均可依序實作，且不影響現有功能。

## 七、是否值得改用 Vue 之分析

### 7.1 目前 Streamlit 涵蓋的範圍

| 區塊 | 行為 | Streamlit 表現 |
|------|------|----------------|
| 對話 | 輸入 → 等整段答案 → 顯示（含圖表/來源/檢索片段） | 每次送問都整頁 rerun，答案是一次性渲染，**沒有 token 逐字串流** |
| 多對話 | 側欄選對話、新對話、清除 | 切換即 `st.rerun()`，狀態在 `st.session_state` |
| 上傳／灌入 | 選檔 → 按鈕 → spinner → success | 流程清楚，但灌入時會卡住整頁 |
| Eval 運行記錄 | 篩選 Tool／關鍵字 → 列表＋展開 | 篩選後 rerun，列表可達數百筆 |
| Eval 批次結果 | 選 Run → 讀 jsonl/metrics → 顯示 | 已有「載入 Run」spinner |
| 圖表 | ECharts option 或 PNG 顯示在 chat 內 | `st_echarts` 靜態渲染，無即時互動（縮放/篩選） |

後端邏輯：`route_and_answer()` 是**同步、非串流**呼叫（Groq/Gemini 可支援 streaming，但本專案目前未用）。

---

### 7.2 Streamlit 的痛點（改用 Vue 會改善的地方）

1. **每次互動都整頁 rerun**  
   切對話、改 TOP_K、送一則訊息，整個 script 重跑，DOM 重繪。對話一多或 Eval 列表長時，體感會頓、捲動可能跳掉。
2. **沒有「回答逐字出現」**  
   若未來要支援 LLM streaming，Streamlit 雖可做（`st.write_stream` 等），但需改後端與前端寫法；Vue + SSE/WebSocket 可自然做逐字更新。
3. **Eval 列表全在伺服器渲染**  
   篩選、排序、分頁都要 rerun；Vue 可前端篩選/虛擬捲動，切換即時、不重載。
4. **圖表僅靜態**  
   ECharts 在 Streamlit 裡是「塞 option 就渲染」，若要 drill-down、連動篩選、即時更新，用 Vue + ECharts 較好控。
5. **部署與擴充**  
   Streamlit 較適合單機/內網；要做多使用者、權限、CDN、PWA，Vue + FastAPI 較易拆前後端與橫向擴充。

---

### 7.3 不值得為「全盤 Vue」重寫的情況

- **定位是內部 Demo / PoC**：Streamlit 已能展示 RAG、多工具、Eval，開發與迭代快，改 Vue 成本高、收益有限。
- **團隊以 Python 為主、無專職前端**：維護 Vue + Node 建置、API 契約、兩套部署，負擔明顯增加。
- **不打算做「回答串流」或「大列表即時篩選」**：目前痛點可透過 Streamlit 小優化（例如 Eval 分頁、限制筆數）緩解。

---

### 7.4 結論：有沒有「值得改 Vue」的地方？

| 情境 | 建議 |
|------|------|
| **維持現狀（Demo / 內部用）** | **不必改 Vue**。繼續用 Streamlit，必要時做低優先優化即可。 |
| **要對外產品化、需串流／大列表／圖表互動** | **有值得改 Vue 的地方**：以「Vue 前端 + FastAPI 後端」重寫 UI，後端沿用 `agent_router`、`rag_graph` 等，只把 `streamlit_app.py` 的「介面＋呼叫邏輯」改成 API；Vue 負責聊天區、多對話、Eval 列表、圖表，並可接 SSE/WebSocket 做逐字輸出。 |
| **折衷** | 先**只在 Streamlit 裡加「回答串流」**（後端改 streaming，前端用 `st.write_stream`），體感提升大、改動範圍小；若之後仍覺得卡，再評估 Vue。 |

**總結**：以「是否值得」來說，**目前專案沒有非用 Vue 不可的理由**；但若目標是產品化、多使用者、或明顯要串流＋大列表＋圖表互動，**就有值得改 Vue 的區塊**——優先順序建議為：**聊天＋串流** > **Eval 列表（前端篩選/虛擬捲動）** > **圖表進階互動**。若維持 Demo 定位，建議**不改 Vue**，把時間放在 Agent 邏輯與 Eval 品質上更划算。
