# 專案分析：頂大研究生 × 業界頂尖工程師 × 產學合作視角

> 以**頂大資工所／研究視角**、**業界 R&D／架構視角**與**產學合作／競賽落地**三維度，對本專案「合約／採購法遵審閱助理」進行分析。供技術決策、作品說明書與產學 PoC 對齊使用。

---

## 一、總覽（一句話）

本專案是 **RAG + 多工具 Agent 路由 + Streamlit** 的競賽導向示範：以 **合約／採購法遵審閱** 為核心場景，整合 LangGraph（檢索→打包→生成）、意圖層路由（合約／法條／Firecrawl／圖表／專家）、Eval 框架與多對話前端，**技術架構完整、競賽對齊度高，具產學 PoC 與短期商業化潛力**；在模型依賴、法條資料來源穩定性與可觀測性上仍有改進空間。

## 二、頂大研究生視角：研究嚴謹度與可重現性

### 2.1 優點

| 面向 | 現狀 | 評價 |
|------|------|------|
| **問題定義** | 合約審閱痛點（耗時、漏風險、需可追溯）→ RAG + 法條查詢 | 清楚、可驗證、與競賽主題一致。 |
| **方法論** | 雙 Prompt（調查員打包證據 → 判官判定）、多查詢檢索、MMR/rerank、Query Rewriting | 有明確的「檢索—生成」分離與可配置開關，利於 ablation。 |
| **Eval** | `eval_set.json`、`eval_set_contract.json`、`run_eval.py`、routing 準確率／latency／tool 成功率 | **可重現的技術驗證**：題集 + 腳本 + 指標，符合學術／競賽「作品完整性」要求。 |
| **文件** | `docs/` 內 RAG 記憶、痛點分析、競賽對齊、合併後分析、專案完整度 | 脈絡完整，有利後續論文撰寫或報告引用。 |
| **檢索範圍** | `chat_id` 過濾、僅搜尋此對話上傳檔案 | 實驗上可做到「單一合約 vs 全庫」對照，利於評估情境化檢索。 |

### 2.2 可強化（研究向）

- **對照實驗**：若要做成論文或深度報告，可補「僅 RAG vs RAG+法條查詢」「strict vs advisor」「有無 multi-query」的對照表（準確率／ hallucination 比例／使用者滿意度）。
- **法條正則與覆蓋率**：`_LAW_REF_PATTERN` 可擴充並記錄「法條字號辨識率」「司法院查詢成功率」，作為技術驗證數據。
- **引用格式**：檢索片段與來源標記（如 `[1]`、`(source#chunk)`）已具備，可考慮統一為學術引用格式（若需對外發表）。
- **模型與版本**：README 或附錄註明 Gemini 型號與 embedding 維度、Pinecone index 設定，方便他人複現環境。

### 2.3 小結（研究生視角）

**嚴謹度**：問題—方法—Eval—文件形成閉環，達競賽與課程專題水準以上。  
**可重現性**：依 README 與 `.env.example` 可重建環境；Eval 題集與腳本可直接用於回歸與報告。  
**建議**：若要往研討會／期刊發展，補齊對照實驗與法條相關指標即可顯著提升說服力。

## 三、業界頂尖工程師視角：架構、可維護性與生產就緒度

### 3.1 架構優點

| 層面 | 說明 |
|------|------|
| **職責分離** | `rag_graph`（檢索與生成）、`agent_router`（意圖與工具執行）、`expert_agents`（領域子 Agent）、`streamlit_app`（UI）邊界清楚，符合單一職責。 |
| **工具註冊** | `SUPPORTED_TOOLS` 集中管理，`_decide_tool` 與執行分支一致，新增 tool 有明確插入點。 |
| **意圖分層** | 嚴格模式 → 合約+法條 intent → Firecrawl intent → 台灣法律 intent → LLM 路由，規則與 LLM 分工明確，可調可測。 |
| **狀態與回傳** | `RAGState` TypedDict、工具統一回傳 `(answer, sources, chunks, tool_name, extra)`，利於擴充與除錯。 |
| **環境與配置** | `.env` + `os.getenv()`，無硬編碼 key；`RAG_*`、`EVAL_*` 等開關齊全，適合不同部署情境。 |

### 3.2 可維護性

- **程式碼**：type hints、logging、註解與文件一致；合併後已處理 `contract_risk_agent` 重複定義與 `contract_risk_with_law_search` 的 `chat_id` 傳遞。
- **依賴**：`pyproject.toml` + uv，Python 3.13+，版本與套件集中管理。
- **測試**：`tests/` 存在；可再增加 router 意圖單元測試、Eval 回歸 CI，降低重構風險。

### 3.3 生產就緒度與風險

| 項目 | 現狀 | 建議 |
|------|------|------|
| **外部 API** | 依賴 Gemini、Pinecone、Tavily、可選 Firecrawl/Groq | 為 Tavily/Gemini 設定 timeout 與重試；考慮 fallback 或降級策略（如法條查詢失敗時仍回傳僅知識庫結果）。 |
| **模型可用性** | 預設 `gemini-3.1-flash-lite-preview` | 若 API 未提供或棄用，需在 README/`.env.example` 註明替代型號（如 `gemini-2.5-flash`）。 |
| **法條來源** | 司法院檢索依 Tavily 搜尋結果 | 若有全國法規資料庫等穩定 API，可整合為主要來源，Tavily 作補強，提升穩定性。 |
| **可觀測性** | `eval_log`、Eval 批次結果、logging | 可補 health check、結構化 log（含 tool、chat_id、latency）、可選 LangSmith/OpenTelemetry 以利維運。 |
| **安全與合規** | 免責聲明、無中國軟硬體 | 可補輸入長度/格式檢查、上傳檔案類型與大小限制、敏感資料處理說明（若走產學或企業 PoC）。 |

### 3.4 小結（業界工程師視角）

**架構**：達業界中上水準，模組化與擴充點清楚，適合做 PoC 或內部工具。  
**生產化**：需補齊逾時/重試、模型 fallback、法條來源穩定性與可觀測性，才較適合對外 SaaS 或高頻企業部署。  
**技術債**：目前不明顯；合併後分析已修復已知問題，維持單一 `contract_risk_agent` 與一致 `chat_id` 傳遞即可。

## 四、產學合作視角：競賽對齊、商業化潛力與落地路徑

### 4.1 競賽對齊（2026 智慧創新大賞）

- **產業應用性（40%）**：目標客群（採購/法務）、價值主張（第一輪審閱縮時、風險可追溯）、社會效益（降低法務負載）已在 `docs/2026智慧創新大賞_競賽須知對齊與專案分析.md` 對齊；建議說明書用 1～2 句可量化敘述（如審閱時間從 1 小時→約 10～15 分鐘）。
- **技術創新性（30%）**：多工具 Agent 路由、合約+法條一條龍、雙 Prompt、Eval 題集與指標，可作為差異化亮點在簡報中強調。
- **作品完整性（30%）**：端到端流程（上傳→灌入→一鍵審閱→檢索片段）、Eval、文件完整度均已具備；3 分鐘影片建議完整走一遍流程並突出「僅知識庫 vs 含法條」。
- **加分項**：開源登錄（GitHub）、AI 輕量化（Gemini Flash、chat_id 檢索範圍控制）可在佐證中簡述。

### 4.2 商業化與產學落地潛力

| 階段 | 內容 |
|------|------|
| **PoC（現狀）** | 單機/團隊內部署、示範合約審閱+法條查詢、Eval 驗證路由與延遲；適合企業 POC、校內展示、競賽決賽。 |
| **試點** | 與一家企業或法務單位合作：限定合約類型（如採購契約、NDA）、限定法規範圍；收集準確率與使用回饋，迭代意圖與 prompt。 |
| **產品化** | 需補：身分與權限、審閱報告匯出（MD/PDF）、可選 on-prem／VPC、合約結構化輸出（如 JSON/表格）供下游系統；法條 API 穩定化。 |

### 4.3 產學合作建議

- **學界**：可將「RAG + 多工具路由在法遵場景的應用」「雙 Prompt 對 hallucination 的影響」作為研討會或產學案技術報告主題；Eval 題集可釋出供社群對比。
- **業界**：適合與法務科技、採購系統整合商、顧問公司談 PoC；強調「單一入口完成合約審閱與法條查詢」「可追溯、可配置、可評估」。
- **競賽後**：若獲獎或受關注，可規劃開源部分模組（如意圖層、Eval 腳本）、保留核心 prompt 或商業邏輯，平衡影響力與商業空間。

### 5.2 短期優先建議（決賽前）

1. **作品說明書與影片**：依 `docs/2026智慧創新大賞_競賽須知對齊與專案分析.md` 與痛點分析，補齊可量化價值敘述與 3 分鐘完整流程示範。

## 六、結論

本專案以 **合約／採購法遵審閱** 為核心，在 **RAG（LangGraph + Pinecone + Gemini）**、**多工具 Agent 路由**（含意圖層與專家）、**合約+法條一條龍** 與 **Eval 框架** 上表現完整且與競賽須知高度對齊，從頂大研究生視角具備足夠嚴謹度與可重現性，從業界工程師視角架構清晰、可維護性佳，從產學合作視角具明確 PoC 與試點潛力。短期聚焦說明書、影片與決賽演示，中長期補齊法條穩定性、可觀測性與產品化項目，即可兼顧競賽成績與產學落地價值。
