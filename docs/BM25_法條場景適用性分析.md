# BM25 在法條／合約審閱專案的適用性分析

> 專案定位為**合約／採購法遵審閱**，知識庫以合約文件為主、並會從合約中抽法條字號再查司法院。本文件分析：是否適合加入 BM25（關鍵字／稀疏檢索），以及實作取捨。

---

## 一、結論先講

| 問題 | 答案 |
|------|------|
| **適不適合做 BM25？** | **適合**。法條／合約領域有大量**精確用語**（條號、法名、契約術語），BM25 能補足純向量檢索在「字面精準匹配」上的不足。 |
| **建議做法** | **Hybrid（向量 + BM25）**：保留現有向量檢索，另加 BM25 檢索後合併結果（例如 RRF），再送後續 dedup／rerank。 |
| **是否必做** | 非必要。若資料量小、問法偏語意（「這份合約有什麼風險」），現有純向量已堪用；若常出現「民法第 184 條」「損害賠償上限」等**條號／術語精準查詢**，加 BM25 效益較明顯。 |

---

## 二、為什麼法條／合約場景適合 BM25

### 2.1 領域特性：精確用語多

| 類型 | 範例 | 說明 |
|------|------|------|
| **條號** | 民法第 184 條、消保法第 7 條、政府採購法第 99 條 | 使用者與文件常**一字不差**，BM25 對「第」「條」「184」等詞的匹配非常有效。 |
| **法規名稱** | 民法、消費者保護法、租賃住宅市場發展及管理條例 | 專案已用 `_LAW_REF_PATTERN` 抽法條字號，同一批詞在檢索時用關鍵字也能精準命中。 |
| **契約術語** | 違約金、管轄法院、損害賠償、給付遲延、保密義務、智財權歸屬 | 合約用語較制式，不同文件常重複相同詞彙，關鍵字匹配可穩定撈到相關段落。 |
| **條款類型** | 付款條件、解除契約、責任上限、爭議解決 | 使用者常直接問「付款條件」「解除契約」，與 chunk 內用字一致時 BM25 表現好。 |

這些都屬於「**問句與文件用字高度重疊**」的情境，正是 BM25 的強項。

### 2.2 純向量可能不足的情境

- 問「**民法第 184 條**」：若 chunk 寫「依民法第 184 條…」，向量通常沒問題；但若 embedding 把重點放在語意而非數字，有時會漏掉「只出現條號」的段落。
- 問「**損害賠償上限**」：多個 chunk 都談損害賠償，但只有一段真正寫「上限」；關鍵字可精準拉高該段排名。
- 問「**違約金**」：合約裡可能同時出現「違約金」「懲罰性違約金」「違約金上限」；BM25 能讓含「違約金」的 chunk 分數明顯高於僅語意相關的段落。

BM25 補的是「**詞有沒有出現、出現幾次**」的資訊，與向量的「語意相似度」互補。

### 2.3 向量仍然重要的情境

- 問法偏**語意／口語**：「如果對方不付錢我可以怎麼求償」→ 需對應到 給付遲延、損害賠償、債權。
- **指代／多輪**：「那下一條呢？」經 query 改寫後，改寫句的語意檢索仍靠向量較穩。
- **交叉概念**：「管轄法院跟雙方登記地址的關係」→ 需同時理解多個概念，向量較擅長。

因此較佳策略是 **兩者並存（Hybrid）**，而不是用 BM25 取代向量。

## 三、Hybrid（向量 + BM25）怎麼做

### 3.1 流程概念

```
使用者問題
    ↓
┌─────────────────────────────────────────┐
│  (可選) Query 改寫（與現有一致)            │
└─────────────────────────────────────────┘
    ↓
┌───────────────────┐    ┌───────────────────┐
│ 向量檢索 (現有)    │    │ BM25 檢索 (新增)   │
│ Gemini embed      │    │ 對 chunk 全文      │
│ → Pinecone query  │    │ 取 top_k          │
└────────┬──────────┘    └────────┬──────────┘
         │                         │
         └──────────┬──────────────┘
                    ↓
           合併結果（例如 RRF）
                    ↓
           依 chunk id 去重、可選 rerank
                    ↓
           後續與現有流程一致（dedup/MMR/rerank → generate）
```

### 3.2 實作選型簡要

| 方式 | 優點 | 缺點 | 適用 |
|------|------|------|------|
| **rank_bm25 + 常駐 chunk 列表** | 無新服務、改動集中在 rag_graph；chunk 可從 ingest 時寫出 JSON 或從 Pinecone metadata 重建 | 資料量大時需常駐或懶載入、BM25 索引需維護 | 中小型、chunk 數可接受（例如數萬以內） |
| **Elasticsearch / OpenSearch** | 內建 BM25、可與 Pinecone 並存，按 chunk id 合併 | 需多一個服務與維運 | 資料量大或已有 ES 的團隊 |
| **Pinecone 稀疏＋稠密** | 單一資料庫、API 支援 sparse-dense hybrid 時可一條龍 | 需確認方案與 API、稀疏向量需在 ingest 時產出 | 若已用或可升級 Pinecone hybrid 方案 |

專案目前僅用 **Pinecone 純向量**，若要加 BM25，實務上多會是「**另一條檢索管道（BM25）＋結果合併**」，而不是把 Pinecone 換掉。

## 四、建議優先級

| 情境 | 建議 |
|------|------|
| **競賽／Demo、資料量不大** | 可維持**純向量**，把心力放在流程與展示；BM25 當**進階優化**。 |
| **已常遇到「查條號／術語不準」** | 建議**實作 Hybrid**：用 rank_bm25 或小型全文索引對現有 chunk 做 BM25，與 Pinecone 結果做 RRF 合併。 |
| **計畫支援大量合約或對外服務** | 可規劃 **Elasticsearch（或 OpenSearch）** 做 BM25，與 Pinecone 並存，長期維護較清晰。 |

## 五、已實作模組（Hybrid 向量 + BM25）

- **rag_common.py**：`load_bm25_corpus`、`save_bm25_corpus`、`append_bm25_corpus`、`build_bm25_index`、`bm25_search`、`merge_hybrid_rrf`；中文以字為單位分詞，利於條號與術語匹配。
- **rag_ingest.py**：灌入時寫入 `bm25_corpus.json`（全量覆寫）；**streamlit_app.py** 上傳灌入時 `append_bm25_corpus`（含 chat_id）。
- **rag_graph.py**：retrieve 節點與 `retrieve_only` 在 `RAG_USE_BM25=1` 時載入語料、跑 BM25、與向量結果做 RRF 合併，再走既有 dedup／rerank。
- **環境變數**：`RAG_USE_BM25=1` 開啟、`RAG_BM25_TOP_K`、`RAG_RRF_K`、`BM25_CORPUS_PATH` 見 `.env.example`。

---

## 六、小結

- 專案**針對法條與合約**，精確用語多，**適合做 BM25**，且以 **Hybrid（向量 + BM25）** 為佳。
- **不是「只能 BM25」**：純向量仍負責語意、改寫後問句與多輪；BM25 負責條號、法名、契約術語的精準匹配。
- **已實作**：以 **rank_bm25** + 語料檔 `bm25_corpus.json` 做 Hybrid，檢索時 RRF 合併；`.env` 設 `RAG_USE_BM25=1` 即可啟用。
