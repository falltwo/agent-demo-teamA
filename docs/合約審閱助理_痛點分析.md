# 合約／採購法遵審閱助理 — 專案痛點分析

> 以產品目標「合約／採購法遵審閱助理」與競賽評分（產業應用性、技術創新性、作品完整性）為軸，盤點目前專案的痛點與改進方向。

---

## 一、產品目標與典型流程（對齊競賽筆記）

- **一句話**：幫企業自動閱讀合約與採購文件，標出風險條款、給出修改建議，並附可追溯的條文出處。
- **典型流程**：  
  1. 使用者選擇「合約審閱」情境  
  2. 上傳合約 PDF／採購文件並灌入知識庫  
  3. 送出審閱請求（固定用語或一鍵觸發）  
  4. 系統產出：條款列表、風險等級、原文節錄、建議改寫、可選法條查詢與風險圖表  
  5. 引用來源可展開、可追溯  

以下痛點依「使用者旅程 → 技術與資料 → 評審與治理」分類。

---

## 二、痛點清單

### 1. 預設「嚴格模式」擋住合約專用工具（高影響）

**現象**：  
- 側欄「嚴格只根據知識庫回答」預設為 **勾選（True）**。  
- 嚴格模式下 `route_and_answer` **不經工具路由**，一律 `run_rag(..., strict=True)`，只走 RAG 的「調查員→判官」流程。  
- 因此 **contract_risk_agent**、**contract_risk_with_law_search** 在預設設定下**永遠不會被呼叫**。

**影響**：  
- 合約審閱助理的核心賣點（合約專家 + 法條查詢 + AI 自檢）在預設情境下用不到，Demo 或評審若未手動取消嚴格模式，會以為「只是一般 RAG 問答」。

**建議**：  
- 將嚴格模式預設改為 **False**，或  
- 依「情境」區分預設：例如側欄增加「合約審閱模式」勾選，當勾選時預設 `strict=False` 並可選「含法條查詢」，一般知識庫問答才預設 `strict=True`。

---

### 2. 無「一鍵合約審閱」、依賴自然語言觸發（中高影響）

**現象**：  
- 競賽筆記期望：「使用者在 Streamlit 前端**選擇合約審閱模式**」或有一鍵觸發。  
- 目前僅能靠使用者**自己輸入**「請審閱這份合約」「合約風險評估並查相關法條」等，觸發邏輯藏在 expander「合約審閱提示」裡。  
- 新使用者容易不知道要打什麼、或打「幫我看這份合約」等變體，可能被路由到一般 `rag_search` 而非合約專家。

**影響**：  
- 作品完整性與應用價值展示不足；評審在 3 分鐘影片或現場 Demo 時若沒說對關鍵字，會看不到合約審閱亮點。

**建議**：  
- 在側欄或對話區上方增加 **「一鍵審閱此對話的合約」** 按鈕（可選「僅知識庫」vs「含法條查詢」），點擊後固定送出預設問題並走對應 tool（如 `contract_risk_agent` / `contract_risk_with_law_search`）。  
- 合約審閱提示改為**預設展開**或放在更顯眼位置（例如空對話時就顯示「上傳合約 → 灌入 → 點擊一鍵審閱或輸入…」）。

---

### 3. 合約輸出未結構化、前端未做條款／風險視覺化（中影響）

**現象**：  
- `contract_risk_agent` 的 Output Format 是 **Markdown 長文**（合約基本資訊、第 X 條、風險等級、修改建議、原文引述），**非 JSON**。  
- 前端僅做 `st.markdown(answer)`，沒有：  
  - 條款清單卡片或表格  
  - 風險等級篩選（高／中／低）  
  - **風險分布圖**（ECharts 長條／圓餅：各風險等級數量、條款類型分布）  
- 競賽筆記與評分「應用價值」期待：條款列表、風險分布圖表、可追溯引用。

**影響**：  
- 產業應用性與作品完整性的說服力不足；法務／採購若要看「高風險有哪些條」需在長文中自己找。

**建議**：  
- **中期**：讓合約專家輸出**可選的結構化欄位**（例如 JSON：條款列表、風險等級、原文片段），前端依此渲染表格 + 篩選 + ECharts 風險分布圖。  
- **短期**：至少在前端偵測到 `tool_name == "contract_risk_agent"` 或 `"contract_risk_with_law_search"` 時，在回答上方加一句「以下為合約風險分析，可展開檢索片段對照原文」，並把「查看檢索片段」預設展開或更顯眼。

---

### 4. 路由歧義：何時走「僅知識庫」vs「含法條查詢」（中影響）

**現象**：  
- **contract_risk_with_law_intent**：僅在「審閱+合約」或「合約+(風險|法條|法律|…)」時觸發 `contract_risk_with_law_search`。  
- 使用者若問「**這份合約有什麼風險**」沒有「審閱」二字，不會進 intent，改由 **LLM 路由**決定；可能選到 `contract_risk_agent`（僅知識庫）或偶爾選錯。  
- 兩種 tool 行為差異大（有無法條查詢、有無 AI 自檢），但使用者無從得知。

**影響**：  
- 同一類問題可能得到不同流程的結果，體驗不一致；評審若試不同問法可能覺得不穩定。

**建議**：  
- 放寬 **contract_risk_with_law_intent**：例如「合約」+「風險」也優先走 `contract_risk_with_law_search`（或至少保證走 `contract_risk_agent`），並在 _decide_tool 的說明中寫清「僅知識庫」vs「含法條查詢」的差異。  
- 或在前端「一鍵審閱」時讓使用者**明確選擇**「僅知識庫」或「含法條查詢」，按鈕對應固定 tool，避免依賴自然語言歧義。

---

### 5. PDF 解析限制（中影響，視合約來源而定）

**現象**：  
- 目前 PDF 僅用 **pypdf** 擷取文字；掃描版 PDF、表格/圖多的合約，擷取率差或為空。  
- 合約常見為掃描件或複雜版型，若上傳後灌入內容很少，檢索與審閱品質會明顯下降。

**影響**：  
- 作品完整性與「使用對象及環境」若寫「支援企業合約 PDF」，評審或實際使用者用掃描版會失望。

**建議**：  
- 在說明書／README 註明「建議使用**文字型 PDF**；掃描版請先以 OCR 轉文字後再上傳」。  
- 未來可考慮整合 **Markitdown / Unstructured** 等，提升表格與掃描版處理。

---

### 6. 空對話／首次使用引導不足（中影響）

**現象**：  
- 空對話時只顯示：「在下方輸入問題，或先展開『為此對話上傳並灌入文件』上傳 .txt/.md/.pdf…」  
- 未強調**合約審閱**流程（上傳合約 → 灌入 → 一鍵審閱或輸入「請審閱這份合約」），合約審閱提示又在側欄 expander 且預設收合。

**影響**：  
- 新使用者或評審第一次進入容易當成「一般問答」用，忽略合約情境。

**建議**：  
- 空對話時增加一句：「**合約審閱**：上傳合約 PDF 後按「灌入」，再輸入「請審閱這份合約」或使用側欄一鍵審閱。」  
- 側欄「合約審閱提示」改為**預設展開**（或首次進入時展開一次）。

---

### 7. Eval 與作品完整性：合約場景的量化不足（中影響）

**現象**：  
- 已有 **eval_set_contract.json**（contract_risk_agent、contract_risk_with_law_search、tw_law_web_search 等），可量測 **routing 準確率**與延遲。  
- 但缺少：  
  - **真實合約樣本**（1～2 份示範合約）與「預期應標為高／中風險」的條款說明；  
  - **合約專用的品質指標**（例如：高風險條款 recall、與人工標註的一致性）。

**影響**：  
- 作品說明書難以寫出「我們以 X 份合約、Y 項高風險條款為基準，系統 recall 為 Z%」等量化驗證，技術創新性與作品完整性說服力不足。

**建議**：  
- 在 `data/` 或 `eval/` 放 1～2 份**去識別化的示範合約**，並在文件或 eval 題集中註明「預期高風險條款類型」（不一定要完整標註，至少列出預期會被抓到的幾類）。  
- 若有餘力，在 `run_eval` 或後處理中對「合約題」增加簡單的關鍵字或結構檢查（例如回答是否含「高風險」「修改建議」等），作為合約場景的輔助指標。

---

### 8. 治理與免責提示不明顯（低～中影響）

**現象**：  
- **contract_risk_with_law_search** 的 system 已要求輸出「本分析僅供內部風險初步檢視與參考，不能視為正式法律意見…」  
- 前端與首屏**沒有**顯著提醒：「本系統為輔助工具，重要合約仍須經律師審閱」。

**影響**：  
- 產業應用性與責任邊界若被評審或客戶追問，缺少「產品層」的明確聲明。

**建議**：  
- 在 Streamlit 側欄或對話區上方加一行小字：「本系統僅供內部初步檢視，**不取代律師審閱**。」  
- 或在「合約審閱提示」內加入同一句免責。

## 四、小結

- 以**合約／採購法遵審閱助理**為目標，目前**最大痛點**是：預設嚴格模式導致合約專用工具從未被使用，以及缺少一鍵審閱與結構化／圖表化輸出，不利 Demo 與評分。  
- 建議優先：**調整嚴格模式預設或情境預設**、**新增一鍵審閱**、**強化合約情境的首屏與提示**；再視時間補上**結構化輸出與風險圖表**、**Eval 合約量化**與**免責聲明**。  
- 本文件可作為產品 backlog 與競賽作品說明書「待改進／未來工作」的對照依據。
